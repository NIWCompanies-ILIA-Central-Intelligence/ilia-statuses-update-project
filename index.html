What we're solving

Right now ILIA mixes three different kinds of status together, so the wrong information can show up in the wrong place — for example, a policy status ("Policy Issued") appearing under Account Status. This tab pulls a client's journey apart into three clear, plain-language questions, each tracked on its own so staff always see the right status in the right spot — here and in Tranche Management.

Dev note: A Disabled account status is not the authoritative status of the account across ILIA. It reflects only the Account track — disabling the account does not override or stand in for the client's Enrollment or Policy status, and other screens/rollups must not treat "Disabled" as the master/headline status for the record.
State machine: Invited → Opened Invite → Account Created → Disabled. Disabled is set manually when an admin disables a client in ILIA — it's not part of the automatic invite→account progression, and can apply from any prior state. (Kyle's set: invited · created · disabled.)

The bug this cleanup fixes: the Account Status column here is rendering a policy value ("Policy Issued") — a policy status, not an account status. After the three-track fix, that column shows account-lifecycle values only (Invited / Opened Invite / Account Created / Disabled), while Policy Status keeps the cleaned policy enum (In-Force, etc.). Struck red = current wrong value; green = corrected. How we make this exhaustive once the repo is connected

Scan for four things and cross-map them — every hit becomes a row above:

  1. Enum literals — INVITED, ENROLLING, PROPOSED, DESIGN, QUOTED, PAYMENTRECEIVED, CASESUBMITTED, DECEASED, IN-FORCE, IN-FORCE-OVERRIDE, SURRENDERED
  2. Field names — policyStatus / accountStatus / enrollmentStatus + variants (this is where cross-wiring shows up)
  3. UI labels — "Account Status", "Policy Status", "Participant Status", "Current Status"
  4. Non-UI consumers — exports, report/statement & email templates, API serializers, downstream integrations, DB migration
